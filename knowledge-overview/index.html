1. Kiến trúc MVC (Model-View-Controller)

SAP UI5 tuân thủ chặt chẽ nguyên lý thiết kế MVC, giúp phân tách rõ ràng giữa giao diện, dữ liệu và logic điều khiển.

  • Model: Đóng vai trò quản lý dữ liệu cho ứng dụng. Nó cung cấp các phương thức để lấy, cập nhật và thao tác với dữ liệu.
  • View: Chịu trách nhiệm định nghĩa giao diện người dùng. Trong UI5, View thường được xây dựng bằng XML.
  • Controller: Là cầu nối giữa View và Model. Nó chứa mã logic (JavaScript) xử lý các sự kiện từ người dùng trên View (như nhấn nút, thay đổi giá trị) và tương tác với Model.

2. Component.js (Thành phần cốt lõi)

Khác với các ứng dụng web thông thường khởi chạy từ index.html, các ứng dụng SAP UI5 chuẩn enterprise (và trên Fiori Launchpad) sử dụng Component.js làm điểm vào (entry point).

Vai trò của Component.js:

  • Tính đóng gói (Encapsulation): Component.js gói gọn toàn bộ ứng dụng thành một thực thể độc lập. Điều này giúp ứng dụng có thể nhúng dễ dàng vào Fiori Launchpad hoặc các ứng dụng khác mà không gây xung đột (ví dụ: xung đột biến toàn cục).
  • Hàm khởi tạo (Init method): Đây là nơi bạn ghi đè hàm init() của lớp cha (UIComponent). Khi ứng dụng khởi chạy, init() sẽ được gọi. Tại đây, Router thường được khởi tạo. 

sap.ui.define([
    "sap/ui/core/UIComponent"
], function (UIComponent) {
    "use strict";

    return UIComponent.extend("my.app.Component", {
        metadata: {
            manifest: "json"
        },
        init: function () {
            // Gọi hàm init của lớp cha
            UIComponent.prototype.init.apply(this, arguments);

            // Khởi tạo router
            this.getRouter().initialize();
        }
    });
});

3. manifest.json (App Descriptor)

manifest.json được xem là "trái tim" cấu hình của ứng dụng. Nó chứa tất cả thông tin metadata và cài đặt của app dưới dạng JSON. Thay vì viết code trong Component.js để khởi tạo Model hay Route, UI5 khuyến khích khai báo tất cả trong manifest.json để tối ưu hóa hiệu suất (khởi tạo bất đồng bộ).

Các phần chính:

  • sap.app: Cung cấp thông tin chung như ID ứng dụng, phiên bản, file ngôn ngữ (i18n), và các data sources (kết nối backend OData).
  • sap.ui: Định nghĩa thông tin giao diện chung như các thiết bị hỗ trợ (desktop, tablet, phone) và icons.
  • sap.ui5: Phần quan trọng nhất, cấu hình chi tiết cho UI5:
    • rootView: View đầu tiên sẽ được load.
    • dependencies: Khai báo các thư viện UI5 cần thiết (như sap.m, sap.ui.core).
    • models: Nơi khai báo và khởi tạo tự động các Model (i18n, OData, v.v.).
    • routing: Cấu hình điều hướng, định nghĩa các routes (đường dẫn URL) và targets (hiển thị view tương ứng).

4. Các loại Model (Model Types)

SAP UI5 có nhiều loại Model nhưng 3 loại dưới đây được sử dụng nhiều nhất:

4.1. JSON Model (Client-side)

Được sử dụng cho dữ liệu lưu trữ phía client (trình duyệt). JSON Model lấy dữ liệu dạng JSON và quản lý nó cục bộ.

  • Đặc điểm: Mặc định hỗ trợ Two-way data binding (Dữ liệu thay đổi trên UI sẽ cập nhật vào Model và ngược lại).
  • Ứng dụng: Quản lý trạng thái cục bộ của view (ví dụ: cờ ẩn/hiện element, trạng thái loading), dữ liệu cấu hình nhỏ, hoặc làm mock data. Không phù hợp với dữ liệu rất lớn vì nó tải toàn bộ vào RAM.

4.2. OData Model (Server-side)

Đây là Model quan trọng nhất trong lập trình doanh nghiệp SAP. Nó được dùng để kết nối và tương tác với các hệ thống backend SAP (như SAP S/4HANA) thông qua giao thức OData (hiện tại hỗ trợ V2 và V4).

  • Đặc điểm: Là Server-side Model. Nó không tải toàn bộ dữ liệu về trình duyệt cùng lúc. OData Model tự động tạo các HTTP request để chỉ tải những dữ liệu cần hiển thị (ví dụ: phân trang / Lazy loading).
  • Sức mạnh: Tự động xử lý các tác vụ sắp xếp (Sorting), lọc (Filtering) và các thao tác CRUD (Create, Read, Update, Delete) trực tiếp dưới database thông qua URL parameters của OData.

4.3. Resource Model (Client-side / i18n)

Dành riêng cho việc quản lý ngôn ngữ và đa quốc gia hóa (Internationalization - i18n).

  • Đặc điểm: Đọc dữ liệu từ các tệp tin .properties (ví dụ: i18n_en.properties, i18n_vi.properties). Nó sử dụng One-time data binding (Ràng buộc một lần) vì các text cố định trên UI không cần thay đổi qua lại liên tục.
  • Ứng dụng: Dịch thuật tất cả tiêu đề, nhãn (label), và văn bản tĩnh trong ứng dụng. Tùy theo ngôn ngữ cài đặt của trình duyệt hoặc User, UI5 sẽ tự tải file ngôn ngữ tương ứng.

5. Data Binding (Ràng buộc dữ liệu)

Có 3 hình thức Binding phổ biến trong XML View:

  • Property Binding: Ràng buộc 1 thuộc tính của UI control với 1 trường trong model (VD: text="{/EmployeeName}").
  • Aggregation Binding: Dùng cho các tập dữ liệu (mảng). Nó tạo ra các template điều khiển lặp lại dựa trên danh sách dữ liệu truyền vào (VD: Gắn vào thuộc tính items="{/Employees}" của một Table hoặc List).
  • Element / Context Binding: Thay vì liên kết từng thuộc tính bằng đường dẫn tuyệt đối, bạn ràng buộc toàn bộ một Layout (như VBox, Form) với một đối tượng cụ thể (VD: {/Employees/0}), lúc này các phần tử bên trong chỉ cần dùng đường dẫn tương đối (VD: {FirstName}).

6. Routing & Navigation (Điều hướng)

SAP UI5 cung cấp hệ thống Routing mạnh mẽ để thay đổi View mà không cần tải lại trang, biến ứng dụng thành một SPA (Single Page Application) thực thụ.

  • Router: Được cấu hình chủ yếu trong manifest.json (phần sap.ui5/routing) và tự động khởi tạo trong Component.js.
  • Route: Định nghĩa một mẫu URL (URL pattern) và trỏ tới một Target tương ứng.
  • Target: Chứa thông tin về View nào sẽ được load và hiển thị khi một Route khớp.

Hệ thống Routing cũng tự động xử lý History (nút Back/Forward của trình duyệt) và cho phép người dùng Bookmarking các URL cụ thể để truy cập thẳng vào một trạng thái của ứng dụng (Deep Linking).
